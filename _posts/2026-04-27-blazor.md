# 📝 **Blazor: три режима рендеринга, их смысл и как они работают**

Blazor Web App в .NET 8/9/10/11 использует **единую компонентную модель**, но поддерживает **три разных режима рендеринга**, которые можно комбинировать на одной странице. Эти режимы определяют, *где* выполняется C#‑код и *кто* отвечает за интерактивность.

---

## **1) SSR — Server‑Side Rendering (статический рендеринг)**
SSR — это когда **сервер выполняет C#‑код и генерирует HTML в момент запроса**.  
HTML не хранится как файл — он создаётся «на лету» и отправляется клиенту.

Особенности:

- HTML **готовый и статичный**.
- Обработчики событий **не подключены**.
- JS Blazor может быть загружен, но **не активен**.
- Поисковики видят **полный HTML**, потому что сервер отдаёт уже отрендерённый документ.
- Страница загружается очень быстро, так как не нужно скачивать WebAssembly.

Документация подтверждает: Blazor «статически отрисовывает HTML‑содержимое с сервера»   [learn.microsoft.com](https://learn.microsoft.com/ru-ru/aspnet/core/blazor/?view=aspnetcore-11.0&utm_source=copilot.com).

---

## **2) SSR Interactive — интерактивная отрисовка на стороне сервера**
Это расширение обычного SSR.

- Сервер рендерит HTML.
- После загрузки сервер **остаётся подключённым** к браузеру через SignalR.
- Все события UI (клики, ввод, изменения) обрабатываются **на сервере**.
- Клиент получает только диффы DOM.

То есть браузер — «экран», сервер — «мозг».

Документация:  
Blazor «поддерживает интерактивную отрисовку на стороне сервера… взаимодействие UI обрабатывается с сервера через подключение в реальном времени»   [learn.microsoft.com](https://learn.microsoft.com/ru-ru/aspnet/core/blazor/?view=aspnetcore-11.0&utm_source=copilot.com).

---

## **3) CSR — Client‑Side Rendering (WebAssembly)**
Это третий, полностью отдельный режим.

- Браузер скачивает .NET runtime (WebAssembly).
- C#‑код выполняется **в браузере**.
- Интерактивность работает без сервера.
- Можно работать офлайн (PWA).

Документация:  
Blazor «поддерживает интерактивность с клиентской отрисовкой (CSR)… зависит от .NET среды выполнения на WebAssembly»   [learn.microsoft.com](https://learn.microsoft.com/ru-ru/aspnet/core/blazor/?view=aspnetcore-11.0&utm_source=copilot.com).

---

# 🧩 Почему SSR называется именно *рендерингом*
Потому что сервер не просто «генерирует строку HTML», а:

- выполняет C#‑код,
- строит дерево компонентов (RenderTree),
- вычисляет состояние,
- формирует визуальное представление UI.

Это аналогично рендерингу в графике:  
из структуры → в отображение.

---

# 🧩 Почему HTML статичен, даже если JS загружен
JS‑файлы могут быть подключены, но **интерактивность не включена**, пока компонент не помечен как интерактивный (`@rendermode`).  
Без этого:

- обработчики событий не подключаются,
- JS Blazor не активируется,
- страница остаётся статичной.

Документация:  
«Содержимое изначально создаётся на сервере и передаётся клиенту **без включения обработчиков событий**»   [learn.microsoft.com](https://learn.microsoft.com/ru-ru/aspnet/core/blazor/?view=aspnetcore-11.0&utm_source=copilot.com).

---

# 🧩 Как поисковики видят HTML, если его нет на сервере
HTML **существует как результат рендеринга**, а не как файл.  
Поисковик делает обычный HTTP‑запрос → сервер рендерит Razor → отдаёт готовый HTML → поисковик его индексирует.

# 📌 Итоговая таблица

| Режим | Где выполняется C# | Где рендерится UI | Интерактивность | Соединение |
|------|---------------------|-------------------|------------------|------------|
| **SSR** | Сервер | Сервер | ❌ Нет | ❌ Не нужно |
| **SSR Interactive** | Сервер | Сервер | ✔ Да | ✔ SignalR |
| **CSR (WebAssembly)** | Браузер | Браузер | ✔ Да | ❌ Не нужно |

# 🎯 Итог в одном предложении
Blazor использует три режима рендеринга: **SSR** (быстрый статический HTML), **SSR Interactive** (сервер управляет UI в реальном времени) и **CSR/WebAssembly** (полностью клиентское приложение), и все они могут сочетаться в одном проекте.
