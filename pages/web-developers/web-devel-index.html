Валидация html-кода, CSS, и ссылок в документе

В контексте материала, валидация - пройти проверку на соответствие стандартам.

Валидация нужна для того, чтобы проверить ваш код на ошибки. Это основная причина, для чего нужно проходить проверку. Также проверка на валидность позволит вам увидеть насколько ваш код соответствует спецификации. Более подробно о том зачем нужна проверка, читайте страницу Why Validate?, на сайте W3C.

В проверки вашего сайта нет ничего сложного. Просто скопируйте адрес вашей страницы и вставьте в валидатор. Или загрузите файл. Процедура настолько простая, что нет смысла давать дополнительные объяснение. Подробности на странице W3C: Руководство пользователя сервиса проверки CSS.

Руководство пользователя сервиса проверки CSS так же применимо и для Nu Html Checker и для Link Checker.

Значок о том что страница успешно прошла валидацию

Если вы решите разметить на странице значок о том что страница прошла валидацию, то в этом случае консорциум говорит о том что значок следует использовать в качестве ссылки для повторной проверки документа.

Подключаем доменное имя к GitHub Pages

Пошаговый план подключения домена к сайту на GitHub Page. По инструкции на странице "Managing a custom domain" (рус. "Управление пользовательским доменом"), из документации по GitHub Pages.

  1. Подтверждение домена. Читать страницу докунтации GitHub Page "Проверка личного домена для GitHub Pages".
  2. Настройка основного(apex)-домена. В инструкции Configuring an apex domain - документации GitHub Pages, всё это дело подробно расписано по полкам. Гляди туда.
  3. Настраиваем поддомен www. Configuring an apex domain and the www subdomain variant (рус. "Настройки основного домена и его поддомена с префиксом «www»")
  4. Настраиваем HTTPS. Документации GitHub Pages Securing your GitHub Pages site with HTTPS (рус. Защита вашего сайта GitHub Pages с помощью HTTPS)

Подтверждение домена. Добавляем TXT запись к домену.

В самой первой статье "Supported custom domains" (рус. "Какие поддерживаются домены"), на странице "About custom domains and GitHub Pages" (рус. "Сведения о личных доменах и страницах") документации GitHub Pages, читаем совет о том что прежде чем настраваить и подключать что-то, сначала нужно подтвердить домен, или если говорить другими словами верифицировать его, привязать к учетной записи GitHub. Это нужно для безопасности, чтобы домен не захватили и не разместили на нем свой сайт. Это не так страшно, но неприятно, и больше может произойти из-за разгильдяйства. Но даже если это и произойдёт, хотя наврятли, то кто то будет хостить свой сайт на вашем домене и всегда можно будет его с этим делом легко обломать, просто изменив у доменного регистратора dns-записи. В общем, если нужны подробности, читай на той же странице статью "Securing the custom domain for your GitHub Pages site" (рус. "Защита пользовательского домена для вашего сайта GitHub Pages").

Нужно добавить новую TXT-запись к домену. Когда вы только приобретаете домен, там уже будет одна запись TXT, её трогать не надо, просто надо добавить ещё одну.

То что GitHub дает для копирования в пункте "Create a TXT record in your DNS configuration for the following hostname" - вставляем в NAME новой TXT-записи. То что во втором пункте "Use this code for the value of the TXT record - вставляем в поле DATA. Ну и готово дело. TXT запись для подтверждения домена для учётной записи GitHub добавлена к домену.

Ждем пока обновится конфигурация DNS. Обычно не долго.

Команда должна показать то, что добавляли в Data. После этой проверки можно нажимать кнопку Verify на GitHub.
